2. BINLOGS:
 *we have seen about the log files and log levels which basically helps us to detect a problem 
 *we could find the bin folder in the application installation path in which we have a log file which has the record of the startup logs

 *these logs are useful until the application startsup

3. VARLOGS:
 *Now the other logs we have is, the varlogs or system logs which is present in the installation path in which 
 *we can configure the levels of logs thru application interface that a user can have 
 *the log levels are default, error, warning, info, debug
 *for minimal log we use default and for extensive log we use debug  
 *we never suggest debug level log for the clients for better performance if the application
 *these varlogs consists of actions that has been taken by th admin, authentication, services etc 
 *these varlogs only maintain last 7days logs of the application 

4. VARLIB
 *In the var folder we have (lib and log)
 *After the log we have seen about the lib folder which contains folders like default, applicationdb
 *Within the default folder we have ssl certificate and pfx of the admin which is initially created for the admin as well as service for the authentication
 *Now in the applicationdb folder we have schema.db and other files that are interlinked 
 *incase of externaldb (ex:snorkel inline) applicationdb consists of schema which is in disc, and data resides in database 
 *incase of internaldb (ex:snorklebx) we have both schema and data in applicationdb
 *while the application is starting up, with the help of schema it loads up required info from the database into the memory  *based on the configuration snorkel application gives access to the backend whenever the request from the end user comes based on the configuration that's been loaded to the memory
 *In return backend gives response to the user through the snorkel
 *Incase if the response page is precursor then snorkle appends the javascript to the response and forwards it to the enduser 
 *if the url is a normal url then it directly sends response to the end user
 
5. BIN:
 *In bin folder we have seen about the applicatioconfig          
 *In the configfile we have seen about the common configurations that are available for all products 
 * we have seen about the cluster architecture that are connected to the same database
 *basically clusters are same applications runnning on different nodes or different systems 
 *The erlang applications starts erlang node whenever the application needs to be started initially inorder to run the application 
 *the node keeps running even the application is closed 
 *every application have unique node name to identify themselves 
 *nodename could be friendlynodename@ip or  friendlynodename@domainname 
 *erlang has epmd(erlang port management daemon) service which runs on 4369 port 
 *every node runs on port that has been configured in the config file 
 *ports in which that the nodes runs are call ddp(datacommunication port)
 *node provide their respective friendlynodename and ddp info to the epmd to communicate with other nodes, this is call internode communication
 *In the config file we have seen about the kernal module where we have configured port range for the nodes inorder to communicate
 *In mnesia module the path location of the applicationdb is specified 
 *erlang have native drivers for mysql so ip and port are enough to specify in config 
 *for other databases dsn name should be specified
